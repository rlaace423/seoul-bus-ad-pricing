# 프로젝트: 서울시 버스정류장 광고가 책정 도구

## 한 줄 정의
도시 구조 정보(유동인구, 공시지가, 주변 상권, 도로망 등)를 입력하면 서울시 버스정류장의 광고 단가를 예측하는 도구를 만든다. 새 위치의 좌표를 넣으면 적정 광고가를 산정해주는 것이 최종 목표. 도시공학과 "데이터기반도시설계" 수업 팀 프로젝트. 평이하고 재현 가능한 방법 위주.

## 문제 정의
- 유형: 회귀(regression). 연속값인 '광고 가격(원)'을 예측한다. 분류 아님.
- 입력(X): 좌표만 있으면 계산 가능한 도시 구조 정보 (아래 피처 목록)
- 출력(Y): 해당 정류장의 월 광고 단가(원)
- 핵심 제약: 신규 위치에도 적용할 도구이므로, '그 정류장의 과거 광고 이력' 같은 신규 위치에서 구할 수 없는 정보는 피처로 쓰지 않는다. 좌표 기반으로 산출 가능한 정보만 사용.

## 도시설계 수업 맥락 (분석의 방향성)
단순 가격 예측에 그치지 않고, "도시의 물리적 구조(가로망/밀도/용도/접근성)가 입지의 상업적 가치를 어떻게 결정하는가"를 보이는 것이 이 수업에서의 의미다. 따라서 모델 성능뿐 아니라 SHAP 등으로 "무엇이 광고가를 끌어올리는가"의 해석을 반드시 포함한다.

## 데이터 (전부 무료/공공 — 매칭 가능하다고 가정하고 진행)
- 광고 가격(Y): 덕플레이스 등에서 수집한 서울 버스정류장별 광고 단가. 정류장 번호로 식별됨.
- 정류장 번호 → 좌표 매칭: 서울 열린데이터광장 (정류소 정보)
- 승하차/유동인구: 서울 열린데이터광장 (버스 정류장별 승하차 인원)
- 공시지가: 국토부 V-World 또는 공공데이터포털
- 도로망: OpenStreetMap (osmnx)
- 상권 POI: 카카오 로컬 API 또는 소상공인시장진흥공단 상가정보
- 지하철역 위치/승하차: 서울 열린데이터광장

* 가장 먼저 할 일: 가격 데이터의 정류장 번호와 열린데이터광장 좌표의 매칭률을 소량 샘플로 테스트할 것. 매칭이 안 되면 이후 전부 무의미하므로 여기부터 확인하고 넘어간다.
* 가격 정규화 주의: 수집된 가격이 양면/단면 기준인지, 제작비·부가세 포함인지 등 조건이 섞여 있을 수 있다. 같은 조건으로 통일한 뒤 Y로 사용한다.

## 피처 목록 (X) — 좌표 기반 산출
우선순위 핵심(먼저 구현): 1, 2, 3, 5번 / 여력 시 추가: 4, 6번

1. 유동인구·교통
   - 정류장 일/월 승하차 인원, 경유 노선 수, 환승 가능 노선 수
   - 가장 가까운 지하철역까지 거리, 해당 역 승하차 규모
2. 토지·부동산 가치
   - 정류장 위치 공시지가, 반경 내 평균 공시지가
   - (가능 시) 용도지역(상업/주거/공업)
3. 상권 밀도 (POI) — 반경 100m / 300m / 500m
   - 반경별 총 상점 수
   - 카테고리별 개수: 음식점, 카페, 병원, 학원, 판매/의류, 금융, 문화시설 등
4. 도로·입지 구조 (여력 시)
   - 인접 도로 등급, 차선 수, 교차로까지 거리, 횡단보도/코너 여부
5. 행정구역·지역 맥락
   - 자치구(범주형), 도심·부도심 중심지까지 거리
6. 도시 형태 지표 (여력 시)
   - 주변 건물 밀도, 평균 건물 높이, 교차로 밀도

### → 최종 사용 데이터 (실제 구현, 2026-06) ⭐
모델 입력 = 좌표산출 도시구조 피처 **31종**(+학습 시 결측플래그 3 = 32) · Y = 광고가(원/면, 2,553정류장). 출처 원장 `data/seoul/raw/README.md`, 피처 사전 `data/seoul/features/README.md`, 산출물 `model/`.

| 분류 | 실제 사용 피처 | 출처 |
|---|---|---|
| **Y 광고가** | 월 광고 단가(원/면) | 덕플레이스 |
| **1 교통** | 버스 승하차_일평균·경유노선수 / 최근접지하철거리·최근접역승하차 | 버스 OA-12912 · 지하철 서울열린데이터 |
| **2 토지** | 개별공시지가(원/㎡, **정류장 점 위치**) | 국토부 V-World |
| **3 상권** | POI 100/300/500m 총수 + 업종 10종(300m) = 13개 | 소상공인 상가(53.7만) |
| **5 지역** | 3도심(시청·강남·여의도) 거리 4종 | 좌표 haversine |
| **4·6 도로/형태** | 교차로수·가로망연장·간선도로연장(300/500m)·간선도로거리 = 7개 | OpenStreetMap(osmnx) |

**계획 대비 미사용/대체 (팀원 추가수집 후보):**
- 공시지가 = **점(point) 값만**, 반경 내 평균 ❌ · **용도지역(상업/주거/공업) ❌ 미수집**.
- 6번 **건물 밀도·높이 ❌**(교차로 밀도만 도로망에 흡수) · 도로 **차선수·횡단보도/코너 ❌**(OSM 부재 → 교차로수·간선도로로 대체).
- **자치구(범주형)·생좌표(위경도)는 모델 피처 아님** — 자치구는 지역CV 그룹·격자 자치구근사로만(위치 자체 암기 방지 → 도시구조 해석·새 지역 일반화 목적).
- ablation 기여순 = 중심지거리 ≫ 승하차·지하철 > POI·공시지가 ≫ 도로망(≈0). → 추가가치는 **용도지역·건물형태** 같은 새 축이 유망(토지·도로 정밀화는 한계기여 낮음).

## 모델
- XGBoost 중심(회귀). baseline으로 선형회귀, 비교군으로 RandomForest.
- tabular 데이터라 딥러닝 불필요.
- baseline 비교 반드시 포함: "전체 평균가격을 답하는 더미 모델" 대비 우리 모델이 얼마나 나은지 보일 것.

## 평가 (회귀 지표 — 정확도/% 아님)
- 주지표: MAE(평균 몇 원 오차), 보조: RMSE, MAPE, R²
- 교차검증(cross-validation)으로 성능 측정. 과적합 점검 목적.
- 공간 데이터이므로 무작위 split뿐 아니라 지역 기반 split(예: 일부 자치구로 학습→다른 자치구로 테스트)도 시도해, 본 적 없는 지역에 대한 일반화를 확인.
- ablation: 유동인구 등 강한 변수를 빼고도 도시구조만으로 어느 정도 예측되는지 확인(어떤 변수가 가격을 만드는지 이해).
- 해석(SHAP)은 두 층위로 산출한다: (1) 전체 모델 차원 — 서울 전반에서 무엇이 가격을 좌우하는가(고정된 경향, 보고서용), (2) 개별 위치 차원 — 각 정류장이 왜 그 가격인지(위치마다 다름, 웹 출력 top_factors용).
- 개별 예측은 점추정뿐 아니라 오차범위(예측구간) 형태로도 제시하는 것을 고려.

## 산출물 — 외부 인계 고려 (격자 사전계산 방식)
웹 프론트엔드는 다른 팀원이 별도 프로젝트로 진행한다. 구현 단순성을 위해 실시간 feature 추출이 아니라 '격자 사전계산' 방식을 채택한다.
- 학습 단계에서 서울 전역을 격자(칸 크기는 정류장 분포 보고 결정)로 나누고, 각 칸의 피처(POI 밀도/공시지가/도로망/접근성 등)를 미리 계산해 테이블로 저장. 무거운 API 호출/공간연산은 이 단계에서 한 번만.
- 추론은 '좌표 → 속한 격자 칸 lookup → 해당 피처로 가격 예측'으로 단순화. 추론 시 외부 API 의존성 0.
- 인계물: ① 학습된 모델(+전처리기) 직렬화 파일(joblib), ② 격자 피처 테이블, ③ 좌표만 받으면 예측가(+오차범위+요인)를 돌려주는 predict 함수, ④ 사용법 README.
- 출력 형태(JSON 예):
  { "predicted_price": 800000, "range": [730000, 870000],
    "top_factors": [ {"name":"유동인구","effect":250000,"direction":"up"}, ... ] }
- top_factors는 '전체 모델의 평균 변수중요도'가 아니라 **해당 위치 하나에 대한 개별 SHAP 기여도**다. 즉 위치마다 다른 값. "이 위치는 유동인구가 많아 +25만원, 상권 밀집으로 +18만원 …" 식으로, 그 좌표가 왜 그 가격인지 설명하는 용도. 웹페이지에서 위치 클릭 시 이 값을 표시할 예정.
- 따라서 격자 칸마다 (예측가격 + 오차범위 + 개별 SHAP 기여 Top N)을 미리 계산해 테이블에 저장한다. 무거운 SHAP 계산도 이 사전계산 단계에서 끝내고, 추론은 lookup만 하도록 한다.
- 참고로 전체 모델 차원의 변수중요도(서울 전반에서 무엇이 가격을 좌우하는가)는 별도로 1회 산출해 보고서/웹 소개용으로 쓴다. 이건 위치별(top_factors)과 다른, 고정된 전체 경향임.
- 우리 범위는 '추론 가능한 모델 + 격자 테이블 + 입출력 계약 + README'까지. REST API 서버/배포는 친구 몫(경계 명확히).

## 보너스 (여력 될 때만, 메인 아님)
학습된 모델의 예측가와 실제가의 차이(잔차)를 이용해, 입지 대비 저평가/고평가된 정류장을 짚어볼 수 있음. 메인 도구가 완성된 뒤 한 챕터로만 다룬다.

## 기술 스택
Python, pandas, geopandas, osmnx, scikit-learn, xgboost, shap. 표준 환경/패키지 관리.

## 한계 (인지하고 진행)
- 수집 가격은 거래가가 아니라 호가일 가능성. 또 대행사가 유동인구 기준으로 가격을 매겼다면, 유동인구로 가격을 예측하는 것은 그 책정 공식을 일부 역산하는 셈일 수 있음. 이 점은 보고서에 명시.
- 서울 내 적용이라 분포 밖 예측 위험은 작지만, 데이터가 적은 외곽 지역 예측은 신뢰도가 낮을 수 있음.

## 미정 사항 (진행 중 함께 정할 것)
1. 분석 단위: 쉘터형 정류장만인지 전체인지 — 가격 데이터 범위 보고 결정.
2. 격자 칸 크기 — 정류장 밀도 분포 보고 결정.
3. 피처 4·6번 포함 여부 — 핵심 피처로 돌려보고 성능 보고 결정.

## 작업 방식 부탁
- 순서: (0) 가격↔좌표 매칭률 테스트 → 데이터 수집 → 정제/가격 정규화 → spatial join → 피처 엔지니어링(핵심 1·2·3·5 먼저) → 모델 학습 → 평가/검증 → 격자 테이블·predict 함수·README.
- 각 단계마다 데이터 분포/중간 결과를 보여주고, 위 '미정 사항' 같은 판단 지점에서는 멈추고 물어봐 줘. 이 판단들은 별도 채팅에서 함께 검토할 수도 있다.
- 나는 머신러닝 초심자다. 새로운 개념이나 선택지가 나오면 간단히 설명을 곁들여줘.
